## Naming Conventions

### Files
- **HTML**: `kebab-case` (e.g., `login.html`, `admin-dashboard.html`)
- **CSS**: `kebab-case` (e.g., `style.css`, `admin.css`)
- **JavaScript**: `kebab-case` (e.g., `firebase-config.js`, `stock-service.js`)
- **Test files**: `kebab-case.spec.js` (e.g., `admin-dashboard.spec.js`)
- **Images**: `kebab-case.jpg` (e.g., `mie-ayam-biasa.jpg`)

### Code
- **Variables**: `camelCase` (e.g., `stockRef`, `CACHE_KEY`)
- **Constants**: `UPPER_SNAKE_CASE` (e.g., `SESSION_KEY`, `VALID_STATUSES`)
- **Functions**: `camelCase` (e.g., `normalizeStockStatus`, `saveStockData`)
- **CSS Classes**: BEM naming (e.g., `.nav__toggle-bar`, `.menu__item-card`)

## State Management

**Pattern**: Local state + Firebase Realtime Database

- **Customer data**: Menu, packages, and stock status from Firebase, cached in localStorage
- **Admin session**: localStorage flag (`admin_logged_in`)
- **No global state management library**: Vanilla JS only

### Data Flow
```
Firebase Realtime DB
        ↓
FirebaseService (firebase-config.js)
        ↓
┌───────────────────────────────────────┐
│ MenuService (menu-service.js)         │
│ PackagesService (packages-service.js) │
│ StockService (stock-service.js)       │
└───────────────────────────────────────┘
        ↓
localStorage Cache
        ↓
UI Updates (main.js for customer, admin scripts for dashboard)
```

## Error Handling

**Pattern**: Graceful degradation with localStorage fallback

1. **Firebase connection failure**: Falls back to cached data in localStorage
2. **Invalid stock status**: Normalizes to "sold_out"
3. **Authentication failure**: Shows error message, prevents login
4. **Network errors**: Displays "Menampilkan data terakhir" indicator
5. **Image upload errors**: Validates type (JPG/PNG/WebP) and size (max 2MB)
6. **Duplicate menu items**: Detected by slug, prevented in MenuService

## Security Model

**Authentication**:
- Admin: Hardcoded credentials (`lariska` / `lariska123`)
- Customer: No authentication required

**Database Rules**:
- Menu data: Public read/write
- Packages data: Public read/write
- Stock data: Public read/write
- No server-side validation (static files only)

**Application-Level Security**:
- Admin write protection via client-side auth check
- Admin page redirect if not authenticated
- Input validation in service layer (max lengths, valid ranges)

## Architecture Constraints

1. **Static files only**: No server-side functions or API endpoints
2. **Firebase client-side SDK**: All database operations from browser
3. **No build process**: Direct HTML/CSS/JS files
4. **No framework**: Vanilla JavaScript only
5. **Mobile-first**: Responsive design with CSS custom properties
6. **IIFE pattern**: All JavaScript modules use IIFE with `'use strict'`
7. **BEM naming**: All CSS classes follow BEM convention
